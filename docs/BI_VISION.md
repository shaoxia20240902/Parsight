# BI 看板：我们想做成什么样（自然语言版）

> 给产品、业务一起对齐用。**开发以 [BI_ADVANCED_OPTIMIZATION.md](./BI_ADVANCED_OPTIMIZATION.md) 为准。**

---

## 一句话

BI 看板不是「把 Excel 画成图」，而是：**先像不同岗位的分析师一样读懂每张表，再站在用户真正关心的角度提出问题，把每个问题都变成一张能查数的图表。**

---

## 用户打开一张表时，我们在后台想什么？

想象一个业务同事刚导入「销售明细」：

1. 他关心的往往不是「总共有多少行」，而是：**卖得怎么样、哪里拖后腿、接下来该盯谁**。
2. 表里可能没有「环比」「留存率」这些列，但**只要原始数据够**，就应该**算出来**再展示。
3. 不同表类型（销售、运营、库存、人力）**关心的 KPI 完全不同**——不能所有表都套「同比、环比」四件套。

所以我们要做的，是模仿「多人读表 → 先定业务场景 → 再提具体问题 → 每题一图」的过程，而不是「看到数字字段就 SUM 一下」。

---

## 整体节奏（从上传到看板）

```text
上传 Excel
  → 入库 + 轻量摘要（快）
  → 后台异步：每张表做「六维理解」（慢，但必须做）
  → 用户可在数据页看理解进度

用户打开 BI / 点击生成看板
  → 检查：每张表是否都有六维理解？
       没有 → 提示「请等待表理解完成」或「去数据页重新生成理解」
       有   → 开始生成看板

看板生成分两条线：
  A. 每个 Sheet 一条线（单表深度分析）
  B. 自定义分类一条线（多表关联成「宽表」再分析，可选）
```

**硬规则（你已确认）**：没有六维理解，**不允许**生成 BI。上传完成后应**自动异步**跑表理解，而不是让用户记得手动点。

---

## A 线：单 Sheet 看板怎么长出来？

对**每一张 Sheet**，走同一套「通用剧本」：

### 第 1 步：读懂这张表

输入不只是字段名，必须包含：

- 六维理解全文（业务含义、粒度、异常、指标口径等）
- 字段画像（哪些是金额、日期、维度、ID）
- **行业猜测**（根据表名、字段名、样本值推断：偏销售 / 运营 / 供应链 / 人力…）

输出一份「**这张表是什么业务**」的短结论，供后面选角色、定场景用。

### 第 1.5 步：Sheet 总览汇总（必有）

在深入各角色之前，先生成 **整张表的经营总览**，回答「这张表整体上是什么规模、核心指标大概多少」：

- 记录数、核心金额/数量指标的 **总量 KPI**
- 若有目标/预算字段：整体 **达成率**
- 若有时间字段：当前周期或最近一期的 **快照**（不是替代趋势分析，而是给用户第一眼锚点）

总览图单独成组，默认 **上板置顶**；与各角色的场景分析互补，不互相替代。

### 第 2 步：选定分析角色（并行，只定人、不出题）

不是固定三个角色，而是 LLM 根据**表类型 + 行业 + 理解内容**，选出 **2～4 个最合适的分析角色**，例如：

| 表像什么 | 可能出现的角色 |
|----------|----------------|
| 销售明细 | 销售负责人、区域经理、财务管控 |
| 用户行为 / 活跃 | 运营增长、留存分析、产品运营 |
| 库存 | 供应链、采购、门店补货 |

每个角色只输出：

- **角色是谁**、**背景说明**（他为什么关心这张表、他的 KPI 语言是什么）
- **不负责**在此步写具体问题（避免角色理解直接跳成问句，导致 SQL 口径飘）

**并行**：多个角色同时选定，互不等待。

### 第 3 步：每个角色先定「业务场景」（并行）

在每个角色下，先写 **1～3 个业务场景**——描述「在什么情况下、要做什么判断」，**还不是**可直接查数的问题。

| 场景（示例） | 说明 |
|--------------|------|
| 月底经营复盘 | 要看各区域/品类贡献是否均衡，有没有过度依赖单一来源 |
| 跟进异常客户 | 要找出近期持续下滑、需要销售介入的对象 |
| 看增长节奏 | 要判断销售额是在加速、放缓还是见顶 |

场景里写清：**决策情境、关心的指标类型、可用的维度/时间**，为下一步出题提供上下文。  
**先有场景，再有问题**——生成 SQL 时能把「这道题在解决什么业务判断」带进去，口径更稳。

### 第 4 步：每个场景下提炼「具体问题」（并行）

在每个场景内，产出 **1～3 个可分析的具体问题**，例如：

- 场景「看增长节奏」→「最近几个月销售额是在加速增长还是见顶回落？」
- 场景「月底经营复盘」→「哪个区域对总销售额的拉动最大？」

LLM 根据**数据里实际有什么**（日期跨度、目标字段等）决定问题类型——**不强制**环比、同比。

### 第 4.5 步：角色级「问题完备性审视」（新增中间层）

**仍在该角色范围内**，不急着画图，先做一次审视：

- 已有场景 + 问题是否覆盖了该角色最关心的 KPI？
- 是否漏了总览类、对比类、异常类、结构类中的某一类？
- 若不足：`need_more_questions: true`，并 **补充 0～N 个新问题**（可挂到已有场景或新建补充场景）

这一层是 **角色对问题的二次把关**，避免场景里只出了一道浅题就进入 SQL。  
补充题与初题合并后，才进入下一步。

### 第 5 步：问题 → 图表（并行）

每个问题变成一张图的配置：

- 业务标题、一句人话问题说明
- 所属角色、**所属场景**（场景 ID 写入 chart，供 SQL 生成器理解上下文）
- 需要计算的 KPI 口径（含表里没有的派生指标）
- 由后端模板生成 SQL，并**执行一次**预览查询

### 第 5.5 步：失败时单次修复（仅针对这一张图）

若出现以下任一情况：

- SQL 执行报错（语法、字段不存在等）
- 查询成功但**结果完全为空**（0 行或 KPI 全为 null）

则交给一个**轻量修复智能体**做一次判断（逻辑不复杂）：

| 诊断结论 | 典型原因 | 修复动作 |
|----------|----------|----------|
| **SQL 有问题** | 字段名写错、聚合/日期写法不适配 TEXT 表 | 在原有问题上**只改 SQL**（或改 `metric_spec` 后重生成 SQL） |
| **问题本身有问题** | 与六维理解/真实字段不符，或业务上这张表算不出来 | **改问题**（收窄口径、换维度/指标，或降低为更可答的一问） |

修复后**再执行一次** SQL 预览：

- **成功且有数据** → 该图进入看板/仓库
- **仍失败或仍无数据** → **放弃这一张图**（本 Sheet 其他图不受影响，不整表失败，不 Mock，**不再第二次重试**）

**原则**：总览汇总 + 各角色经审视后的高价值问题 → 一题一图；能修则修一次，修不好就跳过，质量优先。

## B 线：自定义分类（跨 Sheet）怎么长出来？

只有「多表之间真的有关联、且值得一起分析」时才做。

### 前置条件

- 空间内 **跨表关联** 已生成（或至少有可验证的 JOIN 关系）
- 参与关联的 **每一张表都有六维理解**

### 判断要不要做

LLM 读：关联说明 + 各表理解摘要 + 关联字段重叠情况。

- **不值得做**（表互不相关、关联太弱）→ **0 个**自定义分类，不硬凑。
- **值得做** → 生成 **1～3 个**自定义分类（你已确认的上限）。

### 每个自定义分类是什么？

可以理解成：**把多张 Sheet 在逻辑上拼成一张「宽表」**：

- 明确绑定哪些 Sheet、通过哪些键关联
- 给这个「虚拟宽表」起一个**业务向的分类标题**（如「客户经营全景」「商品动销」）
- 再走与 A 线相同的剧本：总览汇总 → 角色 → 场景 → 问题 → 问题审视 → 图表  
  但 KPI 可以是**跨表的**（例如：客户表 × 销售表的客单价、复购贡献）

不是每个自定义分类都要堆很多 KPI；**1 个高价值主题 + 几张关键图** 即可。

## 通用模板（LLM 填空的结构）

所有 Sheet / 自定义分类共用同一 JSON 骨架，LLM 只填「占位符」内容，不写 SQL：

```text
行业推断 → 业务一句话
  → sheet_summary_kpis[]（总览汇总，必有）
  → perspectives[]（角色 id、名称、背景说明）
       → 每个 perspective.scenarios[]（场景：情境、关注点，不是问题）
            → 每个 scenario.questions[]（具体问题、analysis_type、优先级）
       → 每个 perspective.question_review（是否要补题、补充问题列表）
            → 合并后的全部 questions[]
                 → 后端：每个 question 编译成 chart（带 scenario 上下文）
                 → 执行 SQL；失败/空结果 → 修复智能体（改 SQL 或改问题）→ 再执行一次 → 仍失败则丢弃该图
```

角色数量、场景数量、问题类型都由 **数据和业务** 决定；**场景是 SQL 口径的中间层**，模板保证结构统一、便于并行和验收。

---

## 和「环比 / 同比」的关系

- 它们是 **可选的分析手段之一**，不是默认套餐。
- 是否出环比/同比/纯趋势/达成率，由 LLM 结合：**时间跨度、行业、理解里的业务描述** 判断。
- 后端用**指标模板库**实现；模板条件不满足就**跳过该问题**，并在生成报告里写原因。

## 前端用户会看到什么？

| 情况 | 体验 |
|------|------|
| 表理解还在跑 | BI 页提示等待，可跳转数据页看进度 |
| 表理解失败/缺失 | 提示重新生成理解，不提供空看板 |
| 看板生成中 | SSE 进度：… → 出图与 SQL 校验 →（必要时）单图修复 → 自定义分类 |
| 看板就绪 | 按 Sheet Tab + 少量自定义 Tab 浏览；每张图对应一个「真问题」 |

## 成功标准（业务体感）

1. 打开销售类表：第一眼有 **Sheet 总览 KPI**，再按角色看到场景化深度分析（增长、结构、异常等），而不是只有柱状图排行榜。  
2. 打开运营类表：会出现**留存、活跃、漏斗类**等「表里没列名但业务会问」的题（能算就算，不能算就不出）。  
3. 多表有关联：最多 3 个自定义专题，每张图说得清「跨了哪几张表」。  
4. 所有数字可追溯到 SQL；算不了的题不出现，而不是显示 0。  
5. SQL 报错或空结果时：最多修复重试 **1 次**，仍不行则仅丢弃该图，看板其余图表正常展示。
